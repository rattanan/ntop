# Proposal & Quotation Phase 1 MVP

## Scope

This increment adds Proposal as an additive bounded module and reuses the existing
governed Quote/Approval module. It does not replace legacy Quote read paths or
implement ERP, contract, e-tax, SAP, CRM, CPQ, digital-signature, or billing
behaviour.

Proposal is created from an authorized Opportunity and inherits its Customer.
Every content mutation creates an immutable `ProposalVersion` with ordered
`ProposalSection` records. Status definitions and allowed transitions are stored
as configuration data rather than embedded in application branches. Important
commands write the aggregate, audit event, and idempotency receipt in one
transaction.

AI generation is an optional, human-triggered draft operation. It uses only the
Opportunity, Customer, authorized meeting notes, selected Products, and selected
Template version. Provider/model settings continue to come from the existing
encrypted OpenAI-compatible provider configuration. Provider failure never
blocks manual proposal editing. Generated JSON is strictly validated before a
new editable draft version is persisted.

## Acceptance criteria

- An actor with Proposal manage permission can create a Proposal only from an
  Opportunity visible in the actor's server-side authorization scope.
- Proposal number is generated by a database-backed sequence and Customer is
  inherited from Opportunity; clients cannot substitute another Customer or
  Owner.
- Create, edit, restore, status transition, and AI-generate commands are
  idempotent and append an audit event in the same database transaction as the
  business mutation.
- Each edit or restore creates a new immutable version; a previous version is
  never overwritten and a restored version records its source version.
- Status codes, labels, terminal flags, and allowed transitions are configuration
  data. Invalid transitions and edits to terminal Proposals are rejected by the
  service, not merely hidden by the UI.
- Proposal review is sequential and policy-driven: submission routes to Manager
  review, then Director approval. Required permissions are RolePermissionGrant
  data and maker-checker is enforced by the server for both review decisions.
- A reusable active Template version provides ordered default sections. Template
  data is copied to a Proposal version so later Template edits do not rewrite
  Proposal history.
- AI output contains the required Proposal section codes, has no unknown fields,
  is bounded by input/output validation, and produces an editable Draft version.
  AI outage returns a sanitized availability error and leaves manual workflows
  operational.
- Proposal list/detail/dashboard queries are scoped and bounded to 200 records.
- REST `/api/v1/proposals` create/read/update/restore/status/generate routes keep
  the existing API error envelope, correlation ID, and idempotency conventions.
- Existing Quote creation remains backward compatible. An optional `proposalId`
  may link a new or new-version Quote only when Proposal and Opportunity match;
  the governed Quote calculation, floor-price gates, Approval policy, Decimal
  storage, and server authorization remain unchanged.
- All dates are stored as timezone-aware instants and rendered using the current
  application timezone conventions. Monetary values remain in existing Decimal
  Quote fields; Proposal content does not introduce floating-point money.
- Soft-deleted Proposals are excluded from normal queries. No production path
  uses mock data.

## Phase 1 delivery boundary

The schema keeps clean extension points for document references, delivery events,
and customer acceptance, but this increment does not claim production-ready
binary PDF/Word rendering, email open/download tracking, public customer portal,
object-storage upload/malware scanning, or digital signature. Those require the
approved object storage, queue/worker, mail, public identity, and signing adapters
described in the system architecture. Print uses the authenticated browser view;
Quote export and delivery adapters remain separate follow-on acceptance gates.

## Compatibility and migration

The migration is expand-only: new Proposal tables and nullable `Quote.proposalId`
are added without dropping or changing existing columns. Existing Quote rows and
REST payloads remain valid. Rollback drops only the new foreign key/index/column
and Proposal-owned tables, in dependency order.

## Verification

- Unit: Proposal validation, transitions, immutable version/restore, AI output
  schema, Quote-to-Proposal consistency, and permission denial.
- Contract: REST schemas remain strict while `proposalId` is optional.
- Integration: transaction rollback when audit fails, idempotency receipt replay,
  scoped Opportunity access, and version history against a real database when
  `RUN_DB_INTEGRATION=1`.
- Repository gates: `prisma validate`, lint, TypeScript typecheck, unit tests, and
  configured integration tests.
